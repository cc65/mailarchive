> A makefile is always an option instead of the build in "intelligence" of cl65.
> However, Windows people do usually have problems with makefiles, so it's still

I agree with this smart front end approach. Today I was experimenting
with using cc65 on a Knoppix-system based on the KDevelop suite.

It has almost the same look and feel as a Microsoft Visual Studio. So
it should be easy to work with projects in this environment.
A smart front-end may help to configure KDevelop for Lynx work.

The good thing is that I can use Gimp as my sprite editor and
call "wine handybug" to debug the project.

Actually I have found out the hard way that the Gimp seems to
be the only editor producing the sprites in a format sprpck can
use.

I don't want to see all those intermediate files. So if the
cl65 has a way to do the job it is OK with me.

Once I get the cart-format solved I will start to integrate the
Lynx sprite formats into cl65. (I hope it has the front ends
for reading in bitmaps, palettes and transparency.)
