To add to what Uz said: In general, the CLRCH(N) is not necessary at
all. You would only need it if

1. someone deliberately opened files and set input and output to that
   (CHKIN, CHKOUT in assembly, or CMD x in BASIC)

2. (possibly) if an error occurred in a previous command and the error
   handling code forgot to catch it and close the channels

No. 2 seems very unlikely to me, and no. 1 is exactly what Uz wants to
be able to achieve: Let the user be able to redirect input and output.

Thus, IMHO, the CLRCHN should go completely.

YMMV.
